3. 查看帖子详情

3.1 功能分析

请求参数：帖子ID

响应参数：

  • 该帖子对象 post

  • 帖子作者对象 user

  • 该帖子评论数据列表：List<CommentVo>
    - 评论对象 comment
    - 该评论的作者 user
    - 该评论的点赞数 likeCount

    • 该评论的回复数据列表：List<ReplyVo>
      - 回复对象 reply
      - 该回复的作者 user
      - 该回复的点赞数 likeCount
      - 当前用户对该回复的点赞状态 status（若登录）
      • 当前用户对该评论的点赞状态 status（若登录）
    • 该帖子点赞数 likeCount
    • 当前用户对帖子的点赞状态 status （若登录）

上述响应数据除了点赞数和点赞状态等高并发功能需要从redis中获取之外，其他从数据库获取即可。

业务层需要实现：

  • 根据ID查询帖子对象 DiscussPostService
  • 根据userId查询用户对象 UserService
  • 统计某个实体所获点赞数 LikeService
  • 统计某个用户对某实体的点赞状态 LikeService
  • 根据实体类型和实体ID获取该实体的所有评论对象 CommentService
  • 根据实体类型和实体ID获取该实体的评论总数 CommentService

表现层需要实现：DiscussPostController

  • 封装前端展示数据
  • 实现评论分页

  1. 实现原理：使用Redis存储点赞某个实体的所有userID，这样便可以通过size()得出该实体的点赞数

2.1 事务管理

  • 什么是事务？

    • 事务是由n步数据库操作序列组成的逻辑执行单元，这系列操作要么全执行，要么全放弃执行

  • 事务的特性：

    • 原子性、一致性、持久性
    • 隔离性：针对多事务并发，各个事务的执行互不干扰
      • 常见并发异常：
        • 更新操作：第一类丢失更新、第二类丢失更新
          • 第一类：某一个事务的回滚，导致另一个事务的更新丢失
          • 第二类：某一个事务的提交，导致另一个事务的更新丢失
        • 读取操作：脏读、不可重复读、幻读
          • 脏读：一个事务读到另一个事务未提交的数据
          • 不可重复读：一个事务，对同一个数据前后读取的结果不一致
          • 幻读：一个事务，对用一个表前后查询到的行数不一致

  • 怎么解决？ ==> 事务隔离级别

    • Read Uncommitted：读取未提交的数据。
    • Read Committed：读取已提交的数据。
    • Repeatable Read：可重复读。
    • Serializable：串行化。

  • 实现机制：【参考笔记【期末复习】数据库提纲 (yuque.com)】

    • 悲观锁（数据库）
    • 乐观锁

2.2 spring事务管理

2.2.1 声明式事务管理

  • 利用注解@Transactional进行事务管理

@Transactional 常用配置参数：

  • isolation 事务的隔离级别：

    • public enum Isolation {
    DEFAULT(-1),
    READ_UNCOMMITTED(1), // 最低的隔离级别，可能会导致脏读、幻读或不可重复读
    READ_COMMITTED(2),	//允许读取并发事务已经提交的数据，可以阻止脏读，但是幻读或不可重复读仍有可能发生
    REPEATABLE_READ(4),	//对同一字段的多次读取结果都是一致的，可以阻止脏读和不可重复读，但幻读仍有可能发生。
    SERIALIZABLE(8);	// 最高的隔离级别，所有事务依次逐个执行，但是严重影响性能
    ……
}
